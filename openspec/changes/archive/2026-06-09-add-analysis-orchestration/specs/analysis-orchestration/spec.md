## ADDED Requirements

### Requirement: Injectable Reasoner I/O

The analysis orchestration layer MUST call reasoning through an injectable `Reasoner` protocol. Tests MUST use offline stubs and MUST NOT call a real LLM provider, external network, or require API keys.

#### Scenario: Offline reasoner stub drives analysis
- **WHEN** analysis runs in tests with stub author and reviewer reasoners
- **THEN** all reasoning responses come from the injected stubs without any external LLM I/O

#### Scenario: Production provider is outside this change
- **WHEN** a caller needs a real LLM provider
- **THEN** this change exposes only the protocol boundary and does not require provider credentials

### Requirement: Three-step Thesis Orchestration

The analysis orchestration layer MUST run thesis creation in this order: free generation by an author reasoner, completeness critique, then adversarial falsification by a reviewer reasoner. The completeness critique MUST record an audit object and MUST NOT rewrite the free-form thesis body. The adversarial reviewer MUST be independent from the author by instance id and persona.

#### Scenario: Complete three-step flow produces a confirmed thesis candidate
- **WHEN** author and reviewer reasoners return valid free-generation, critique, and adversarial outputs
- **THEN** the orchestrator assembles a confirmed thesis with `body`, `completeness_critique`, `adversarial_falsification`, `track_record`, confidence, and source signal ids

#### Scenario: Completeness critique preserves body
- **WHEN** the critique step runs after free generation
- **THEN** the recorded `completeness_critique.body_unchanged` is true and the original `body` is preserved

#### Scenario: Self-review is rejected before confirmation
- **WHEN** the reviewer has the same instance id or persona as the author
- **THEN** the orchestrator MUST NOT confirm or persist the thesis

### Requirement: Contracted Thesis Persistence

Confirmed theses produced by analysis orchestration MUST be persisted only through `ContractStore.add_thesis()`. The orchestration layer MUST NOT write directly to thesis storage or bypass `thesis-contract` validation.

#### Scenario: Confirmed thesis is stored through ContractStore
- **WHEN** orchestration successfully assembles a thesis
- **THEN** it calls `ContractStore.add_thesis()` and returns the stored thesis id

#### Scenario: Missing critique cannot confirm
- **WHEN** the assembled thesis lacks `completeness_critique`
- **THEN** `thesis-contract` validation rejects it and the orchestrator does not report success

#### Scenario: Missing adversarial review cannot confirm
- **WHEN** the assembled thesis lacks `adversarial_falsification`
- **THEN** `thesis-contract` validation rejects it and the orchestrator does not report success

### Requirement: Minimal MVP Boundaries

The analysis orchestration layer MUST NOT implement production LLM provider calls, signal clustering, target generation, or feedback calibration in this change. Those concerns MUST remain behind explicit future-change boundaries.

#### Scenario: Analysis does not create targets
- **WHEN** a thesis is successfully created by analysis orchestration
- **THEN** no target record is generated by this capability

#### Scenario: Analysis does not calibrate feedback
- **WHEN** a thesis is successfully created by analysis orchestration
- **THEN** no calibration signal or feedback score is generated by this capability
